## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser admin UI → PocketBase REST API | Authenticated admin requests cross here; PocketBase collection rules are the real enforcement layer |
| Admin search input → PocketBase filter string | Untrusted text from the search box is interpolated into a filter expression |
| npm registry → build | Third-party package (`qrcode.react`) introduced into the dependency tree |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-FILTER | Tampering | search input → getList filter | mitigate | Use `pb.filter('(studentName ~ {:q} || certificateCode ~ {:q})', { q })` for all user-supplied search text; never string-concatenate into the filter |
| T-03-XSS | Tampering/Information Disclosure | certificate field values rendered in table | mitigate | Render every field with React JSX `{value}`; no `dangerouslySetInnerHTML` anywhere in the list |
| T-03-AUTHZ | Elevation of Privilege | /admin list fetch | accept | PocketBase `listRule = @request.auth.id != ""` (INFRA-06) enforces auth server-side; AdminGuard is defense-in-depth for UX only |
